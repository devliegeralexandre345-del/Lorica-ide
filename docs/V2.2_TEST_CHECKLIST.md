# Lorica v2.2 — Manual Test Checklist

Before shipping v2.2, walk through this list. Each test has three parts:

- **Setup** — what to prepare before the test
- **Do** — exact steps to perform
- **Expect** — what success looks like (concrete, observable)
- **Fail signs** — what indicates the feature is broken

Priority flags:
- 🔴 **P0** — ship-blocker if broken
- 🟠 **P1** — important, not ship-blocker
- 🟡 **P2** — polish feature, minor glitches acceptable

Realistic total on one OS: **~90 minutes**. Single-platform minimum (🔴 only): **~20 minutes**.

---

## How to open the dev console (for error checking)

Most tests below ask you to check for errors in the dev console. To open it:

- **Windows/Linux**: `Ctrl+Shift+I` while Lorica is focused
- **macOS**: `Cmd+Opt+I`

Red entries = real errors. Yellow = warnings (usually OK to ignore). Keep the **Console** tab open during all P0 tests.

## 🔴 Smoke test — 5 min - "ca lance" 

### 1.1 App launch
- **Setup**: Fresh terminal, be in project dir (or use shortcut)
- **Do**: Run `npm run tauri:dev` (or launch the installed MSI)
- **Expect**: A Lorica window opens within ~5 seconds, welcome tab is shown, no red errors in the dev console, no red toast
- **Fail signs**: Window doesn't open; opens then closes; infinite splash; red console error mentioning "panic", "undefined", "Failed to fetch"

### 1.2 Open folder
- **Setup**: Have any folder on disk with a few files in it (e.g. this project's repo)
- **Do**: Press `Ctrl+O` OR click `File → Open Folder`. Pick the folder.
- **Expect**: File tree appears on the left with the folder's contents, window title shows the folder name, a toast says "Project opened" or similar
- **Fail signs**: Dialog doesn't open, folder loads empty even though it has files, app freezes for >10s

### 1.3 Open and edit a file
- **Setup**: File tree populated (from 1.2)
- **Do**: Click any text file in the tree (e.g. `README.md`). Type a few characters at the end.
- **Expect**: A tab opens with the file's contents, title shows a dot (•) indicating unsaved changes after typing
- **Fail signs**: Click does nothing; blank editor; characters don't appear when typing

### 1.4 Save
- **Setup**: File from 1.3 has unsaved changes
- **Do**: Press `Ctrl+S`
- **Expect**: Dot (•) disappears from the tab title, toast says "Saved" (or similar)
- **Verify externally**: Open the same file in Notepad / `cat` / another editor → your change is there
- **Fail signs**: No visible feedback; error toast; external read shows old content

### 1.5 Terminal
- **Setup**: Project open
- **Do**: Press `` Ctrl+` `` (backtick). Type `echo hello` and press Enter.
- **Expect**: Terminal panel slides up from the bottom; prompt appears; after Enter, "hello" is printed on the next line
- **Fail signs**: Nothing happens; terminal opens but accepts no input; output garbled; colors wrong

### 1.6 Session restore
- **Setup**: Have at least 2 tabs open and a project loaded
- **Do**: Close Lorica completely (`Ctrl+W` on the window, or `Ctrl+Shift+Q`). Reopen it.
- **Expect**: The same project, the same tabs (in the same order), the same active tab, same theme
- **Fail signs**: Back to welcome tab; project not remembered; only some tabs restored

## 🔴 Vault & Security — 5 min

### 2.1 First vault init
- **Setup**: Fresh profile (delete `%APPDATA%/com.lorica.app` / `~/Library/Application Support/com.lorica.app` / `~/.config/com.lorica.app` to simulate first-run)
- **Do**: Launch Lorica. Navigate to `Settings → Security → Initialize Vault` (or similar). Enter a password, confirm.
- **Expect**: "Vault initialized" toast; status indicator shows "Unlocked"
- **Verify externally**: A file appears at the appdata vault location (usually `vault.bin`). Opened in a hex editor, it looks like random bytes (not readable JSON)
- **Fail signs**: Password input silently accepts empty; vault file is plain-text JSON; error message is cryptic

### 2.2 Lock / unlock cycle
- **Do**: From Settings or the lock icon in the status bar, click Lock Vault. Then try to access a feature that needs the vault (e.g. view stored secrets).
- **Expect**: A re-entry password prompt appears
- **Do**: Enter the **wrong** password first
- **Expect**: A clear error "Invalid password" (not a crash, not a generic "something went wrong")
- **Do**: Now enter the **right** password
- **Expect**: Features unlock normally

### 2.3 Secret storage - "Ne l'as pas save dans le vault puis comme launch en tauri:dev ne sauvegarde pas en local apres le relaunch"
- **Setup**: Vault unlocked
- **Do**: `Settings → AI → Enter API key` (use a fake `sk-test-123456` for testing). Click Save.
- **Expect**: Toast "Saved". The input field either clears or shows masked dots.
- **Do**: Close Lorica completely. Reopen it. Unlock vault. Go back to Settings → AI.
- **Expect**: The key is still stored (shown as masked dots), retrievable
- **Fail signs**: Key stored in plain localStorage (check dev console → Application → Local Storage); key lost on restart

## 🔴 Git integration — 10 min - "ne m'as pas dmd mes identifiant ducoup le commit ne fonctionne pas sinon le reste a l'air de fonctionner"

### Setup for this section
Open a folder that is a real git repo (any git project works, ideally the Lorica repo itself).

### 3.1 Status panel
- **Do**: Open the Git panel (`Ctrl+Shift+G` or sidebar icon)
- **Expect**: See current branch name at the top, a list of modified/untracked files grouped into "Unstaged" and "Staged" sections
- **Fail signs**: Panel empty despite changes; branch shows "unknown"; error toast about git

### 3.2 Stage / unstage
- **Setup**: Make sure you have at least one unstaged change (edit a file, don't save → actually save, so git sees it)
- **Do**: Click the `+` icon next to an unstaged file
- **Expect**: File moves from "Unstaged" to "Staged" within 1 second
- **Do**: Click the `-` icon next to the now-staged file
- **Expect**: Moves back to "Unstaged"
- **Verify externally**: Run `git status` in a terminal → matches what the UI showed

### 3.3 Commit
- **Setup**: At least one staged change
- **Do**: Type a commit message in the input, press Commit
- **Expect**: Toast "Committed", working tree clears to "nothing to commit", the commit appears at the top of the log with your message
- **Fail signs**: Commits silently without message; UI shows commit but `git log` in terminal doesn't; wrong author

### 3.4 Diff view
- **Setup**: At least one modified file (not yet staged)
- **Do**: Click a modified file in the git panel
- **Expect**: A diff view opens showing added/removed lines with `+` / `-` gutters, color-coded
- **Fail signs**: Diff empty; diff shows whole file as added; lines misaligned

### 3.5 Live Blame
- **Do**: Open any committed file, press `Ctrl+Alt+B`
- **Expect**: Gutter shows author name + commit hash + relative date for each line. Uncommitted lines (if any) have a different color/indicator.
- **Fail signs**: Gutter stays empty; shows "?"; all lines show the same person even for files with history

### 3.6 Log view
- **Do**: In the git panel, click on "History" or "Log" tab
- **Expect**: A scrollable list of recent commits, each with message, author, date
- **Fail signs**: Empty list on a real repo; dates in epoch seconds instead of readable format; crashes on a 1000-commit repo

**⚠ Known requirement**: `git` must be on PATH. If not, expect a clear error "git not found" — not a white screen.

## 🟠 Run tab — one check per language (15 min) - "fonctionne pour tout"

### Setup for this section

For each language, create a minimal test file. You can use `/tmp/lorica-test/` (Linux/WSL) or `C:\Users\devli\lorica-test\` (Windows).

### 4.1 Python
- **Setup**: Create `test.py` with:
  ```python
  print("hello from python", 2 + 2)
  ```
- **Do**: Open the file in Lorica. Open Run panel. Language auto-detects as Python. Click Run.
- **Expect**: Output panel shows `hello from python 4` and "Exited with code 0"
- **Fail signs**: "Python not found" (install from python.org); weird stdout encoding issues; doesn't terminate

### 4.2 JavaScript
- **Setup**: Create `test.js` with:
  ```js
  console.log("hello from node", 2 + 2);
  ```
- **Do**: Open, Run
- **Expect**: Output `hello from node 4`, exit 0
- **Fail signs**: "node not found"

### 4.3 TypeScript (⚠ needs `tsx` or `ts-node` globally)
- **Setup**: First run `npm i -g tsx` in a terminal. Then create `test.ts`:
  ```ts
  const n: number = 42;
  console.log("hello from ts", n);
  ```
- **Do**: Open, Run
- **Expect**: Output `hello from ts 42`
- **Fail signs**: If tsx/ts-node NOT installed, you should see an **actionable error** "Install tsx: `npm i -g tsx`" (not a cryptic SyntaxError)

### 4.4 C (⚠ needs gcc)
- **Setup**: Create `test.c`:
  ```c
  #include <stdio.h>
  int main(void) { puts("hello from c"); return 0; }
  ```
- **Do**: Open, Run
- **Expect**: Output `hello from c`, exit 0
- **Fail signs**: "gcc not found" → install MinGW-w64 (Windows) or `build-essential` (Linux) or Xcode CLT (macOS). Clear install hint should be shown.

### 4.5 C++ (⚠ needs g++) — this is the one that was broken
- **Setup**: Create `test.cpp`:
  ```cpp
  #include <iostream>
  int main() { std::cout << "hello from cpp\n"; return 0; }
  ```
- **Do**: Open the file. Click Run.
- **Expect**: Compilation succeeds silently, then output `hello from cpp`, exit 0
- **Fail signs**: "Compilation failed" with gcc errors about `-std=c++17`; compile hangs; output empty; exit -1 with no message

### 4.6 Rust (needs a Cargo project)
- **Setup**: In a terminal: `cargo new lorica-test-rust && cd lorica-test-rust`. Open this folder in Lorica, open `src/main.rs`.
- **Do**: Click Run
- **Expect**: First run downloads/compiles, then output "Hello, world!", exit 0
- **Fail signs**: "cargo not found"; runs but no output; crashes the dev server

### 4.7 Go (needs Go toolchain)
- **Setup**: Create `test.go`:
  ```go
  package main
  import "fmt"
  func main() { fmt.Println("hello from go") }
  ```
- **Do**: Open, Run
- **Expect**: Output `hello from go`
- **Fail signs**: "go not found"; compile errors

### 4.8 C# (needs dotnet + a project)
- **Setup**: In a terminal: `dotnet new console -o lorica-test-cs && cd lorica-test-cs`. Open this folder in Lorica.
- **Do**: Open `Program.cs`. Run.
- **Expect**: "Hello, World!" in output
- **Fail signs**: "dotnet not found"

## 🟠 Debug tab (DAP) — 20 min - "fonctionne"

### Setup for this section

Debug requires language-specific adapters installed. The app should show **actionable install messages** when adapters are missing. Verify the messages themselves even if you don't install everything.

### 5.1 Python debugger
- **Setup**: `pip install debugpy` in a terminal, then restart Lorica
- **Do**:
  1. Open `test.py` (from 4.1)
  2. Click in the gutter next to `print(...)` to set a breakpoint (a red dot appears)
  3. Switch to Debug tab, click Start Debug
- **Expect**: Debug session starts, execution pauses at the breakpoint, Variables panel shows local vars, Call stack shows `<module>` frame
- **Do**: Click Continue
- **Expect**: Program finishes, "Debug session terminated" toast
- **Fail signs**: Start Debug does nothing; session starts but breakpoint never hits; adapter error with no install hint

### 5.2 C++ debugger (the previously broken one)
- **Setup**: Install `lldb-dap` (Linux: `sudo apt install lldb-14` then symlink/alias, OR install LLVM 18+ which ships it). On macOS: `brew install llvm` gives you `/opt/homebrew/opt/llvm/bin/lldb-dap`. On Windows: pain, use WSL for this.
- **Do**: Open `test.cpp`. Set a breakpoint on the `std::cout` line. Start Debug.
- **Expect**: The binary is compiled (with `-g` for debug symbols), debugger attaches, breakpoint hits
- **Fail signs**: "lldb-dap not found" **with clear install hint** (acceptable if missing); session starts but breakpoint never hits; debugger can't read source lines

### 5.3 Go debugger
- **Setup**: `go install github.com/go-delve/delve/cmd/dlv@latest`, ensure `$HOME/go/bin` is on PATH
- **Do**: Open `test.go`, set a breakpoint on `fmt.Println`, Start Debug
- **Expect**: Delve launches, breakpoint hits, variables readable

### 5.4 Missing adapter → actionable error
- **Do**: Remove any installed Python DAP (or test with a language you don't have), try to Debug
- **Expect**: A toast or output panel message like "No debug adapter found for 'python'. Install debugpy: `python -m pip install debugpy`"
- **Fail signs**: Silent failure; cryptic "No such file or directory"; app freezes

### 5.5 Debug controls work once paused
For a paused session (any language):
- **Do**: Click Step Over, Step In, Step Out in sequence
- **Expect**: Execution advances correctly at each click, variables update
- **Do**: Click Terminate
- **Expect**: Session closes, UI returns to non-debug state

## 🟠 LSP (autocomplete + diagnostics) — 10 min "fonctionne mais pas assez complet et ca ne propose pas les librairies quand on utilise include en cpp ou autres languages"

### Setup
Each language needs its server installed:
- Python: `pipx install 'python-lsp-server[all]'`
- Rust: `rustup component add rust-analyzer`
- TS/JS: `npm i -g typescript typescript-language-server`
- C++: `sudo apt install clangd` (Linux) or `brew install llvm` (macOS) or it ships with LLVM on Windows

### 6.1 Autocompletion
- **Do**: In a `.py` file, type `import os` on one line, then start typing `os.pa` on the next
- **Expect**: A dropdown shows `path`, `pathsep`, etc. Arrow keys navigate, Enter/Tab accepts
- **Fail signs**: No dropdown; empty dropdown; "LSP server not found" toast with **install hint** (acceptable if not installed)

### 6.2 Diagnostics
- **Do**: In a `.py` file, type `undefined_func()` on a line by itself
- **Expect**: Red squiggly under `undefined_func`, hover shows "undefined name 'undefined_func'"
- **Fail signs**: No squiggly; wrong message; app freezes trying to run LSP

### 6.3 Hover
- **Do**: Hover over `print` in a Python file
- **Expect**: A tooltip appears with the function signature and docstring
- **Fail signs**: No tooltip; shows raw error; Tooltip appears at wrong coordinates

### 6.4 Go to definition
- **Do**: Place cursor on a function call (e.g. `os.path.join(...)` in Python), press `F12`
- **Expect**: Jump to the definition in a new tab or scroll to it in the current
- **Fail signs**: Does nothing; jumps to wrong location; opens an unrelated file

## 🟠 AI features — 10 min - "Fonctionne depuis longtemps"

### Setup
You need either an Anthropic API key (https://console.anthropic.com/) or a DeepSeek key (https://platform.deepseek.com/). Use a minimum-credit test account.

### 7.1 First-run RGPD consent
- **Setup**: Start from a clean state — `localStorage.clear()` in dev console if needed to reset consent
- **Do**: Settings → AI → paste API key → save. Then try to send a message to Copilot ("hello")
- **Expect**: On the FIRST AI call, a consent modal appears with text about data being sent to Anthropic/DeepSeek, with "Accepter" and "Refuser" buttons
- **Do**: Click "Refuser"
- **Expect**: Modal closes, a message appears like "Consentement refusé. Aucune donnée envoyée.", no network request goes out (verify in dev console → Network tab)
- **Do**: Try again, this time click "Accepter"
- **Expect**: Modal disappears, message sends, response arrives

### 7.2 Consent persistence
- **Do**: After accepting once, close Lorica. Reopen. Send another AI message.
- **Expect**: No consent modal (the choice persists in localStorage under `lorica.ai.consent.v1`)
- **Fail signs**: Modal reappears; modal never goes away; network request fires before consent given

### 7.3 Copilot basic
- **Do**: In Copilot panel, type "Write a hello world in Rust"
- **Expect**: Response streams in, shows a code block with Rust hello world
- **Fail signs**: No response; error toast "Invalid API key"; response is garbage encoding

### 7.4 Inline AI Edit
- **Do**: Select a few lines of code in the editor, press `Ctrl+K`, type "add error handling", submit
- **Expect**: Diff preview appears, showing the AI's proposed replacement. Accept/reject buttons.
- **Fail signs**: `Ctrl+K` does nothing; preview shows empty; reject leaves a mess

### 7.5 Key removal
- **Do**: Settings → AI → clear the API key, save. Try to use Copilot.
- **Expect**: Clean error "Configure your API key in Settings first", no request fires
- **Fail signs**: Sends request with empty Authorization header; silent failure

## 🟠 Semantic search — 5 min - "Fonctionne"

### 8.1 Index build
- **Setup**: Open any project (at least a dozen source files)
- **Do**: Click the semantic search icon or `Ctrl+P` then `?` prefix. Enter a query like "authentication logic".
- **Expect**: First time: prompt "Build semantic index?" → click Yes → progress bar for 10-60 seconds (depends on project size), then results appear
- **Verify externally**: File `<project>/.lorica/semantic.bin` exists after the build (use file explorer)
- **Fail signs**: Build hangs; crashes during embedding; index written as empty file; model download fails

### 8.2 Search quality
- **Do**: With the index built, try a natural language query about your code
- **Expect**: Top results are actually relevant to the query, with code snippets shown
- **Fail signs**: Random results; same result for every query; crashes on query

### 8.3 Incremental rebuild
- **Do**: Edit one file in the project, save it. Trigger a re-index or wait for auto.
- **Expect**: Re-index is fast (1-3 seconds for one file change), not a full rebuild
- **Fail signs**: Takes as long as the initial build; doesn't pick up the change at all

### 8.4 Clear index
- **Do**: Find the "Clear index" button or menu
- **Expect**: The `.lorica/semantic.bin` file is deleted, next search re-prompts to build
- **Fail signs**: Button does nothing; file not deleted

## 🟠 Terminal — 5 min - "fonctionne"

### 9.1 Basic I/O
- **Do**: `` Ctrl+` ``, then type `echo lorica-test` and Enter
- **Expect**: "lorica-test" printed
- **Fail signs**: Keypresses don't register; output garbled; \r\n issues making everything on one line

### 9.2 Resize
- **Do**: Grab the top of the terminal panel and drag up/down to resize
- **Expect**: Terminal content reflows, no scroll lock, `COLUMNS` and `LINES` env vars update (test with `echo $COLUMNS` on Unix)
- **Fail signs**: Rows/cols stuck; overflow visible; text clipped

### 9.3 Multiple tabs
- **Do**: Click "+" to create a second terminal tab. Type different commands in each.
- **Expect**: Two independent shell instances, switching tabs preserves each one's history
- **Fail signs**: Both tabs show same output; crashes on second tab

### 9.4 Long-running + kill
- **Do**: Start `ping -t google.com` (Windows) or `ping google.com` (Unix). After a few lines, close the tab with X.
- **Expect**: The ping process is killed (verify with Task Manager / `ps` / `htop`)
- **Fail signs**: Process survives tab close, zombie processes accumulate

### 9.5 Cross-shell
- **Do**: On Windows, open a terminal and run `powershell -Command "Get-Date"`. On Unix, run `bash -c "uname -a"`.
- **Expect**: Correct output from the spawned shell
- **Fail signs**: "Command not found"; wrong shell environment

## 🟠 File watcher — 2 min - "ne mets pas a jour automatiquement je dois refresh moi meme"

### 10.1 External file creation
- **Setup**: Project open in Lorica
- **Do**: In Explorer/Finder/another terminal, create a new file at the project root (e.g. `touch /path/to/project/new-file.txt`)
- **Expect**: Within 1-2 seconds, the file tree in Lorica refreshes and `new-file.txt` appears (no manual refresh needed)
- **Fail signs**: File never appears without manual refresh; appears after 30+ seconds

### 10.2 External file deletion
- **Do**: Delete a file outside Lorica (while its tab is open in Lorica)
- **Expect**: Tab shows an indicator that the file is gone (e.g. grayed out, marked "external delete")
- **Fail signs**: Saving to the deleted file silently re-creates it without notice; crash on save

### 10.3 Noisy directories filtered
- **Do**: Run `npm install` in the project, or `cargo build`, or create 100 files in `node_modules/`
- **Expect**: The file tree does NOT refresh constantly. The dev console does NOT show a flood of `fs:change` events.
- **Fail signs**: UI freezes during npm install; flood of events; tree updates 100 times per second
