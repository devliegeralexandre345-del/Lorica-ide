# Lorica v2.2 — Ship Plan

Breaking down the outstanding "📋 Ce qu'il faut" column into concrete,
estimated, prioritized work. Three phases:

- **Phase A — Ship v2.2** — polish items that block a clean release
- **Phase B — v2.2.x patches** — bug fixes caught during testing
- **Phase C — v2.3 roadmap** — architectural work too big for v2.2

Each item has a **priority** (P0/P1/P2), an **estimated effort**, and a
**blocker status**. Estimates assume solo work by the maintainer who
knows the codebase.

## Phase C — v2.3 roadmap (4-8 weeks of work)

### C1. 🟡 IDE plus léger (performance for low-end PCs) — 1 week
**Priority:** P1 for v2.3
**The problem:** main.bundle + vendors + codemirror ≈ 1.7 MiB. Cold start
is OK but RAM can hit 400-600 MB with all features enabled. On a modest
laptop that's painful alongside a Node dev server and a browser.

**Plan (measurable, not hand-wavy):**

1. **Baseline** — profile current startup with Performance HUD:
   - Time to first paint
   - Time to welcome tab interactive
   - RAM after 5 min idle with a project open
   - RAM during typing in a 2000-line file
   Record in `docs/perf-baseline.md`.

2. **Kill the worst offenders:**
   - Audit all `useEffect` that run on every keystroke
   - Move all AI features behind explicit activation (only load the
     code when user toggles it on)
   - Virtualize the file tree (currently renders the whole tree)
   - Virtualize the Git log (currently renders all commits)

3. **Bundle split:**
   - Make Pyodide truly lazy (only fetch when user runs Python in
     Sandbox)
   - Lazy-load Mammoth (already lazy — verify)
   - Consider splitting CodeMirror language parsers so only the current
     file's parser loads

4. **Startup:**
   - Defer non-essential hooks (`useHeatmap`, `useSemanticAuto`) until
     the user actually opens their respective panels

**Acceptance:** cold startup under 2.5s, idle RAM under 250 MB.

### C2. 🟡 Extensions system (architectural) — 2-3 weeks
**Priority:** P2 (nice-to-have)
**Context:** User says "réduire les features de base → faire des
extensions à la place". This is a major architectural change.

**What it needs:**
- Extension manifest format (`.lorica-ext.json`)
- Extension loader (dynamic import at runtime)
- API surface exposed to extensions (editor, commands, panels, events)
- Sandboxing (extensions shouldn't have direct filesystem access
  beyond their declared scope)
- Extension store UI (list, install, uninstall, update)
- Extract 3-5 existing features as reference extensions (e.g. Focus
  Timer, Scratchpad, Regex Builder — the standalone ones are easiest)
- Docs for extension authors

**Realistic scope:** 2-3 weeks for a v1 extension API that's stable
enough to depend on. Easy to under-estimate — extension APIs are
notoriously hard to design because they're effectively permanent (you
can't break them without burning every extension author).

**Recommendation:** Do NOT rush this for v2.3. Spec it in v2.3, ship it
in v2.4.

### C3. 🟡 Research for new features — ongoing

## Ongoing — Maintenance plan

### M1. Dependency hygiene
- **Quarterly:** `cargo update` + `npm outdated`, review changelogs,
  upgrade majors one at a time
- **Monthly:** `cargo audit` + `npm audit` for security advisories
- **Per PR:** `npm run build` + (once SAC sorted) `cargo check`

### M2. Testing rhythm
- Every commit: `npm run build` passes
- Every tag: full `V2.2_TEST_CHECKLIST.md` run on at least 2 OSes
- Every quarter: write one new integration test for the highest-risk
  feature (currently: agent tool-use end-to-end, or vault encryption
  round-trip)

### M3. Bug triage SLA
- 🔴 Crash / data loss / security: acknowledge <24h, patch <72h
- 🟠 Feature broken for many users: <1 week
- 🟡 UX nits / edge cases: batch into next minor

### M4. Commit discipline
- One logical change per commit
- Commits signed with user.name/email (the bug we just fixed!)
- Conventional commit prefixes (`fix:`, `feat:`, `perf:`, `docs:`,
  `refactor:`, `test:`, `chore:`) — optional but makes CHANGELOG
  generation trivial

### M5. Code health
- When a module exceeds 600 lines or 3 responsibilities → split it
- When a bug is fixed, ask: "is there a whole class of bugs like this?"
  (e.g. the TimeScrubBar hooks bug → I already grep-hunted for similar
  patterns)
- Quarterly dead-code pass: `cargo +nightly udeps`, manual audit of
  unused frontend exports

## Honest notes

- **Don't try to land C1 (perf) and C2 (extensions) in v2.2.** Both are
  week+ projects and would push v2.2 back by a month. Ship v2.2 with
  current perf (good enough for most machines), plan v2.3 around perf,
  plan v2.4 around extensions.
- **The signing situation won't change without traction.** Every
  v2.2.x patch is a chance to mention SignPath in release notes so
  that when we re-apply, there's a paper trail of sustained shipping.
- **The checklist I wrote is not optional.** I don't have confidence
  there aren't more categorical bugs until someone (you) runs through
  it once. The bugs you've already found (vault not persisted, git
  identity, watcher not refreshing, settings not persisted) are exactly
  the kind of thing static audit misses.
