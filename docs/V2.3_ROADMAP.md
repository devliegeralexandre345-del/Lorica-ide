# Lorica v2.3 — Roadmap (research draft)

_Drafted 2026-04-25. Subject to change after v2.2 user feedback._

This is a research-backed scoping doc, not a commitment. It captures
candidate features for v2.3 derived from a competitor scan (VS Code,
Cursor, Zed, Windsurf) and a community pain-point pass. Every entry
cites a source. Effort sizes use the rubric in `V2.2_SHIP_PLAN.md`:
**S** = 1–3 days, **M** = 1–2 weeks, **L** = 3+ weeks.

## Tracks

Already locked in from `V2.2_SHIP_PLAN.md` Phase C:

- **C1 — IDE plus léger** (perf, low-end laptops). Two passes done in
  v2.2: main bundle 989 KiB → 284 KiB, idle hooks via
  `requestIdleCallback`, lazy panels. **v2.3 target:** cold start
  < 2.5 s, idle RAM < 250 MB on a 2-core / 8 GB machine. Measurement
  baseline lives in the Performance HUD; record numbers in
  `docs/perf-baseline.md` before any further changes.
- **C2 — Extensions system v1**. v2.2 shipped a "soft" feature catalog
  (28 toggles, `utils/features.js`). The plan recommends specifying the
  dynamic-import API in v2.3 and shipping it in v2.4. v2.3 deliverable
  is a draft `EXTENSION_API.md` and one reference extension extracted
  from the catalog (Focus Timer is the easiest candidate — no shared
  state, single panel).
- **C4 — Code signing**. Re-apply to SignPath after v2.2 traction
  (target: 100 stars, 2-3 external contributors). No engineering work,
  just paperwork.

C3 (research) is folded into this document.

## Candidate features for v2.3

### High-value, low-effort (target 4-6 of these for v2.3)

| Feature | Source / who has it | Effort | Why it matters | Risk |
|---|---|---|---|---|
| **Multi-line search & replace** | Zed v0.230.0 (Apr 2026), long-standing JetBrains/IntelliJ user complaint | **S** | Cross-IDE pain point; current `GlobalSearch.jsx` is single-line. Common need for codebase-wide regex transforms. | Low. CodeMirror search panel already supports it; UI work only. |
| **Git status decorations in file tree** | Zed v0.229.0 (Mar 2026); VS Code has had this since 2017 | **S** | Visual signal for unstaged/staged/untracked files without opening Git panel. Lorica has Git Blame gutter but no tree decorations. | Low. Backend already exposes `git status`; map result to file tree icons. |
| **Git graph / commit history visualization** | Zed v0.231.1 (Apr 2026) | **M** | Visual branch/commit topology is one of the top "missing" complaints in non-JetBrains editors. Lorica has Git Panel but no graph view. | Low. Pure rendering on top of `git log --graph` output. |
| **Floating / detachable editor windows** | VS Code v1.100 (Apr 2025) | **M** | Multi-monitor users want to drag a tab into its own window. Plausible in Tauri (multi-window API exists). | Medium. Tauri multi-window state sync is non-trivial; scope to read-only floating preview first. |
| **Staged-changes gutter decorations** | VS Code v1.100 (Apr 2025) | **S** | Distinguishes "what's staged" from "what's modified" in the gutter without opening Git panel. Lorica gutter currently shows only modified lines. | Low. Extends the existing Git decorations layer. |
| **Inline AI co-author commit trailer** | VS Code v1.110 (Feb 2026) | **S** | Auto-append `Co-authored-by: Claude/Copilot` when commit contains AI-generated edits. Trivial UX win, GDPR-clean (local config toggle). | Low. Hook into existing commit flow; opt-in. |
| **Reusable prompt files / custom instructions** | VS Code v1.100 (Apr 2025), Cursor `.cursorrules` | **S** | `.lorica/prompts/*.md` and `.lorica/instructions.md` auto-attached to agent context. Lorica has Project Brain but no per-project prompt templates. | Low. Filesystem read + concat into agent system prompt. |
| **AI-assisted merge conflict resolution** | VS Code v1.105 (Sep 2025), Zed v0.228.0 (Mar 2026) | **S** | "Resolve with AI" button on `<<<<<<<` markers. High-frequency pain (every rebase). | Low. New action that opens existing agent panel with conflict context pre-loaded. |
| **`@branch-diff` agent context mention** | Zed v0.228.0 (Mar 2026) | **S** | Mention `@diff` in the agent panel to attach the full branch diff vs. main. Replaces the manual "paste this diff" flow. | Low. Reuses existing diff command output. |

### Medium-value, medium-effort (target 2-3)

| Feature | Source / who has it | Effort | Why it matters | Risk |
|---|---|---|---|---|
| **Native devcontainer support (read-only first pass)** | Zed v0.231.1 (Apr 2026) | **M** | Tauri can shell out to `docker exec`. v1 = "open this folder in this container's shell" without full Remote-Containers parity. | Medium. Matches a top VS Code Remote feature request; dev container spec is stable. |
| **Native insert-mode autocomplete polish** | Neovim 0.12 (Mar 2026) ships built-in LSP autocomplete; competitor parity | **M** | Lorica already has LSP + 52k static entries. Gaps: ranking by recency, fuzzy match on `detail`, snippet-template insertion. Low-flash visible improvement. | Low. Tweaks to existing `staticSource`/`lspSource` code. |
| **Voice / speech-to-text input for the agent** | Cursor "Upgraded Voice Input" (Apr 2026) | **M** | Web Speech API is free in Tauri's webview on Windows/macOS. Optional, opt-in, GDPR-clean if local-only (Web Speech is on-device on macOS / via Edge speech on Windows). Linux needs fallback. | Medium. Cross-platform parity is annoying; ship Windows + macOS first. |
| **Agent worktree / "background task on a branch"** | Cursor (Apr 2026), Windsurf (Dec 2024), Zed v0.233.5 (Apr 2026) | **M** | Run an agent on a feature branch in `git worktree` without disturbing the current editor state. Lorica already has Swarm Dev (parallel agents) but no worktree isolation. | Medium. Worktree management is fiddly on Windows (case-insensitive paths). |
| **MCP server marketplace inside Extensions panel** | VS Code v1.105 (Sep 2025) | **M** | Lorica's agent tool layer is in-tree; surfacing an MCP `@mcp` filter in the existing Extensions registry is a small, real win for power users. | Low-medium. Need an MCP transport (stdio + sse); spec is stable. |

### Aspirational / v2.4+ (defer)

| Feature | Source / who has it | Effort | Why we're NOT doing it now |
|---|---|---|---|
| **Real-time collaborative editing (CRDT, like Zed)** | Zed core feature | **L** | Hard problem. Requires CRDT engine, presence server, identity. We have no server-side infra and no plan to run one. **Dropping for v2.3.** Reconsider only if there's strong demand and a viable peer-to-peer route (Yjs over WebRTC). |
| **Cloud agents / "Devin in IDE"** | Windsurf 2.0 (Apr 2026), Cursor background agents | **L** | Requires running our own cloud, plus a sandbox / VM layer. Out of scope for a privacy-first single-binary IDE. |
| **Tiled / parallel agent panes** | Cursor (Apr 2026), Zed v0.233.5 | **L** | Lorica has Swarm Dev (parallel) but the UI today is sequential. Real tiled UI is a layout-engine rewrite. Defer until extensions system lands. |
| **Full extensions system v1 (dynamic import + sandboxed API)** | C2 from v2.2 ship plan | **L** | Already explicitly slated for v2.4. v2.3 spec only. |
| **Interactive canvases / dashboard responses** | Cursor v3 (Apr 2026) | **L** | Cool but speculative. Only one competitor has it, with mixed reception. Not a daily-frequency feature. Wait for signal. |
| **Agentic browser tool (autonomous web app verification)** | VS Code v1.110 (Feb 2026), v1.115 | **L** | Requires headless browser embedding + per-site action ledger. Heavy lift, narrow audience (web devs only). Revisit in v2.5. |
| **Native screen-sharing on Linux Wayland** | Zed v0.230.0 (Apr 2026) | **M** | Useful only with collaborative editing — see above. Drop with collab. |
| **BYOK aggregator (OpenRouter / Ollama / Google)** | VS Code v1.117 (Apr 2026) | **M** | Lorica already supports custom API keys per provider. Aggregator UI is polish, not a v2.3 priority. Track as v2.3.x patch. |

## Open questions

1. **Perf budget enforcement.** C1 says "cold start < 2.5 s, idle RAM
   < 250 MB". Do we want a CI gate that fails the build if a PR pushes
   the main bundle past 320 KiB? Cheap to add (`size-limit`), and
   prevents perf regression by accident.
2. **Voice input — Linux.** Web Speech API doesn't work on most Linux
   webviews. Ship voice on Win+Mac only, or block until Linux parity?
3. **Devcontainer scope.** Read-only "open shell in container" is a few
   days. Full Remote-Containers parity (LSP/DAP forwarding, file editing
   inside the container) is weeks. v2.3 should pick the read-only first
   pass; confirm with users before scoping more.
4. **MCP transport.** stdio-only is enough for local servers. Adding
   SSE/HTTP transport is one more week and unlocks remote MCP. Is the
   audience there?
5. **Extension API surface.** Before C2 spec lands, we need to decide
   which capabilities are exposed: panel registration, command
   registration, command-palette entries, editor decorations, agent
   tools. The reference extension (Focus Timer) needs at minimum panel
   + command + settings.

## Suggested v2.3 shortlist (working set, ~5 weeks of work)

If we pick 6 from "high-value low-effort" + 2 from "medium" + the C1
perf pass:

1. C1 perf pass (target met) — 1 week
2. Multi-line search & replace — 2 days
3. Git status decorations in file tree — 2 days
4. Git graph view — 1 week
5. Floating editor windows — 1 week
6. Inline AI co-author trailer — 1 day
7. Reusable prompt files (`.lorica/prompts/`) — 2 days
8. AI-assisted merge conflict resolution — 2 days
9. `@branch-diff` agent mention — 1 day
10. Native autocomplete polish (ranking/snippets) — 1 week
11. C2 extensions API draft + Focus Timer reference — 1 week (spec only)

Total: ~5 weeks. Leaves room for inevitable v2.2.x patches + user
feedback bugs uncovered post-launch.
